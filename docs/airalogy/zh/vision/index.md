# Airalogy 作为通用 AI-ready 数据和智能中枢的蓝图

Airalogy 的长期目标是建设面向 AI 的通用数据和智能中枢：不限定单一数据类型、单一应用场景或单一组织层级，而是连接不同场景和不同系统中的数据，保留原始来源，建立统一身份、统一引用、统一关系、统一来源链，并通过 Protocol 和智能运行时将数据转化为可验证、可追溯、可自动处理、可被 AI 直接使用的上下文。

这个蓝图定义 Airalogy 未来要建设的核心方向。面对实验数据、邮件、即时聊天、会议记录、文件、任务、代码、传感器数据和业务系统数据，Airalogy 将通过协同、桥接、结构化和智能调度，把分散数据组织成统一的数据和智能网络。

Airalogy 将服务于组织级的数据体系建设。一个公司、研究机构、大学、实验室或跨组织协作网络，都会产生分布在不同层级、不同部门、不同软件和不同流程中的数据。Airalogy 要把组织结构、人员、项目、课题、实验、会议、任务、文件、沟通记录、业务系统和外部数据源系统性地组织起来，让这些数据不只是被保存，而是能够被管理、追踪、关联、调用和智能利用。当整个组织体系的数据被 Airalogy 统一组织后，AI 赋能不再只是单点工具能力，而会成为支撑组织运行、科研探索、业务决策和知识生产的智能中枢。

Airalogy 可以在多个层级上使用。对于个人，它可以成为个人数据、知识、任务、文件、沟通记录和 AI 助手的智能中枢；对于团队、实验室、公司、大学或研究机构，它可以上升为组织级的数据和智能中枢，统一管理组织结构、权限边界、项目网络、知识资产、业务流程和智能 agent。个人层级强调个人记忆、个人生产力和个人智能增强；组织层级强调多人协作、组织知识沉淀、跨系统数据治理、流程自动化和面向组织目标的主脑运行时（Masterbrain Runtime）。

## 核心判断

未来的 AI 系统不只是需要更多数据，而是需要 ready data。所谓 ready data，至少应满足以下条件：

- 可寻址：每个对象都有稳定 ID，能够被引用、检索和关联。
- 可追溯：每个结构化结果都能追溯到原始消息、文件、记录、会议或外部系统对象。
- 可解释：数据的结构、含义、约束、采集方式和处理规则是显式的。
- 可验证：关键字段可以被 schema、Protocol、hash、版本和人工确认机制校验。
- 可关联：数据之间的关系可以被系统维护，而不是只存在于人的记忆中。
- 可执行：AI 不只是读取数据，还能基于 Protocol 记录或生成 Airalogy Record、摘要、任务、决策、检查结果和后续流程。

因此，Airalogy 的通用性来自一个核心目标：未来一切重要数据都可以成为 Airalogy Record，或者至少能够被 Record 化，并在主脑中以统一的 Airalogy Record 语义被理解、推理、调用和治理。Airalogy 要建设的不是又一个把所有数据粗暴塞进同一数据库的系统，而是一套让任意来源数据都能被引用、保留来源链、经过 Protocol 约束、进入智能运行时的通用框架（universal framework）。

这里需要区分物理存储和语义使用。Airalogy 的目标不是“把所有数据都物理物化成 Airalogy Record”，而是让所有重要数据都能进入 Airalogy 的统一语义网络，并在主脑使用时可以以 Airalogy Record 或 Record 化视图的形式被调用。Airalogy Record 应当被严格理解为基于 Protocol 记录、校验、生成或交换的数据对象；聊天消息、邮件、文件、会议和外部系统对象不需要被强行改造成实体 Record，它们可以作为 Message、Email、File、Meeting 或 SourceObjectRef 存在，并在被 Protocol 采集、确认、转化或派生时形成 Airalogy Record 或 Record 化视图。

## 主脑（Masterbrain）愿景

Airalogy 更长远的终极目标，是成为主脑的平台基础。这里的主脑不是普通聊天助手，也不是单个工具调用 agent，而是在最高目标、意志或使命驱动下，能够占据智能闭环中心位置的系统：它理解目标，组织数据，提出假设，设计方案，调度工具和助手，分析结果，形成结论，并把结论转化为后续行动。

这个愿景不要求目标一定来自人。目标、使命、策略和边界可以来自个人、组织、公司治理结构、科研计划，也可以来自 AI 自主生成的探索方向。随着 AI 能力增强，AI 不只是执行任务的助手，也可能成为能够主动设定方向、组织资源、调度执行并承担经营或科研决策的中心智能。站在公司角度，它可能表现为 AI CEO；站在科研组织角度，它可能表现为 AI PI 或 AI 主脑科学家。

Airalogy 平台不只是一个被 AI 查询的资料库，而是主脑形成所需的基础设施。它让主脑能够拥有长期记忆、数据来源、对象关系、Protocol 规则、工具边界、执行历史和可追溯证据。没有这层基础设施，AI 很容易停留在一次性对话或孤立工具调用；有了这层基础设施，AI 才可能从“回答问题”走向“持续理解、持续规划、持续执行、持续复盘”。

这意味着 Airalogy 的主脑愿景可以覆盖多个阶段：在较保守的形态中，人或组织提出目标，AI 主脑负责规划和调度；在更激进的形态中，AI 本身也可以提出目标、启动项目、调度资源并驱动后续行动。

Airalogy 承载主脑：它把分散数据组织成可被智能使用的世界模型，并通过 Protocol、connector、agent 和工作流调度各种 Assistant。Assistant 可以是人、软件工具、外部 SaaS、自动化设备、机器人或其他 AI agent。

因此，Airalogy 的终局不是做一个更大的数据库，而是建设一个能让主脑运转起来的数据和智能操作系统。

## 分层架构

Airalogy 作为数据和智能中枢可以分为五层：

```text
Raw Data Store
  Airalogy Record，即通过 Airalogy Protocol 直接记录、校验或生成的实验记录、观察、表单、检查结果和人工确认；以及原始文件、原始消息、原始邮件、原始仪器数据和原始导出包，不轻易改写。

Canonical Object Layer
  把内外部数据映射成 Organization、Unit、Team、Lab、Project、Person、Message、Thread、Email、Meeting、Document、File、Task、Record 等统一对象；其中 Record 是基于 Protocol 记录、校验、生成或交换的数据对象。

Protocol / Schema / Automation Layer
  定义数据、交互和行动如何被采集、解释、校验、转换、摘要、派生、执行和反馈。

Relation / Timeline / Index Layer
  维护对象关系、时间线、全文索引、向量索引、来源链和权限感知的检索视图。

Application / Agent Layer
  为搜索、问答、实验记录、会议纪要、项目管理、知识库和自动化 agent 提供 ready context。
```

其中 Raw Data Store 首先承载 Airalogy 最有特色的原生数据：基于 Protocol 形成的 Airalogy Record。Raw Data Store 和专门系统通常是事实来源。Raw Data Store 不只是外部数据的归档位置；当实验、观察、会议、检查或人工确认直接通过 Airalogy Protocol 记录时，Airalogy 本身也是这些数据的出生地和事实来源。Canonical Object Layer 是统一对象层，负责让不同来源的数据进入同一套 Airalogy 对象模型；Relation、Search、Vector 等索引层应当是可重建的派生层。Airalogy 的核心价值在于统一语义、引用、规则、来源链和智能调度，而不是替代所有底层存储。

在组织级场景中，Airalogy 需要表达和管理完整的组织层次。公司可以有事业部、部门、团队、项目和业务流程；大学可以有学院、实验室、课题组、合作项目和教学科研活动；研究机构可以有平台、课题、样本、实验、仪器和数据产出链。Airalogy 的数据和智能中枢要把这些组织层级与具体数据对象连接起来，使组织能够从任意层级追踪数据来源、理解上下文、调用知识、复盘过程并驱动下一步行动。

## 专门数据继续专门存储

不同类型的数据天然有不同的结构和访问模式。即时聊天适合消息事件流，邮件适合 MIME/thread 模型，会议适合音视频、转写和纪要并存，实验数据可能需要 Airalogy Record、表格、仪器文件和样本关系，时间序列数据可能需要专门的 time-series database，文件和附件适合 object storage。

Airalogy 将保留这些数据的专门结构，并在其上建立统一引用、关系和智能处理层。核心设计原则是：

```text
专业系统负责存得对
Airalogy 负责找得到、连得上、解释得了、能自动处理
```

例如，微信或飞书消息可以继续存在消息数据库或平台 API 中，Airalogy 只保存统一引用、索引、关系和派生结果。Protocol 在需要处理聊天时，通过引用请求 connector 读取消息范围，再生成会议纪要、任务、决策、摘要或 evidence-linked Airalogy Record。

## 与成熟专业软件协同

Airalogy 将与飞书、微信、Slack、邮箱、会议软件、网盘等成熟专业软件协同，而不是在平台内重复建设它们已经成熟的交互能力。成熟专业软件负责实时协作、消息收发、组织权限、通知、移动端体验、会议接入和文档编辑；Airalogy 负责把这些系统产生的数据接入数据和智能中枢。

更清晰的边界是：

```text
飞书 / 微信 / 邮箱 / 会议软件 / 仪器软件
  负责数据发生、日常交互、专业存储和原生工作流。

Airalogy
  负责统一引用、跨系统索引、Protocol 处理、AI 结构化、来源链、派生任务、知识网络和主脑运行时（Masterbrain Runtime）。
```

因此，Airalogy 不替代成熟的专业软件，而是通过统一引用、连接器、Protocol 和来源链，将分散在专业软件中的数据组织成 AI-ready 的统一数据网络。

例如用户仍然可以在飞书群里讨论实验方案，但 Airalogy 可以读取被授权的消息范围，抽取实验条件、行动项和决策，生成 Airalogy Record 草稿、Task、Decision 或 Annotation，并将这些结果关联到项目、Protocol、文件和后续实验结果。这里飞书仍然负责沟通体验，Airalogy 负责让沟通产生的数据进入智能中枢。

## 数据和智能中枢

Airalogy 不必成为所有数据的出生地，也不必物理保存所有原始数据，但它应当成为所有重要数据进入智能系统之前的语义中枢。外部系统可以负责产生数据，Airalogy 负责理解数据、连接数据、治理数据和激活数据。

这不意味着 Airalogy 不能产生原始数据。当用户直接通过 Airalogy Protocol 记录实验过程、观察结果、会议纪要、检查结果或人工确认时，这些记录本身就是 Airalogy Record，也可以称为 Airalogy-native source data。它们不是外部数据的摘要，也不是二次派生结果，而是由 Airalogy 原生采集形成的原始数据，并应当进入 Raw Data Store 和来源链。

作为数据和智能中枢，Airalogy 平台需要逐步掌握以下能力：

- 统一身份：为重要对象建立稳定的 Airalogy ID 或外部 SourceObjectRef。
- 统一引用：让 Protocol、agent、搜索和 UI 都能用同一方式引用外部对象和内部对象。
- 统一索引：在权限允许的前提下建立全文索引、向量索引、时间线索引和关系索引。
- 统一关系图：把人、项目、实验、会议、聊天、文件、任务、决策和 Airalogy Record 连接起来。
- 统一来源链：让每个 AI 输出、Airalogy Record、任务和决策都能追溯到原始证据或 Protocol 采集过程。
- 统一 Protocol 处理：用 Protocol 定义数据如何被解释、校验、转换、摘要和派生。
- 统一 AI 上下文装配：让 agent 获得权限感知、来源明确、结构稳定的上下文。
- 统一自动化入口：让 AI 和工作流可以基于这些数据执行后续动作。
- 统一主脑运行时：让主脑能够在长期记忆、工具调用、Protocol、来源链和任务执行之间形成闭环。

用户不一定需要在 Airalogy 中重新聊天或重新开会，但应该可以在 Airalogy 中追问：“上周关于项目 A 的讨论最终确定了哪些实验条件？这些条件有没有进入正式 Protocol？谁负责执行？对应的 Airalogy Record 在哪里？有哪些消息、文件和会议纪要支持这个结论？”这类问题体现的正是数据和智能中枢的价值。

从主脑视角看，这类问题不是简单搜索，而是一次主脑式推理：系统需要识别项目、定位跨系统证据、理解 Protocol 状态、追踪负责人、判断任务进展，并把结果组织成可行动的下一步。

## Airalogy Source Bridge

为了让分布在不同系统中的数据可以被 Airalogy 统筹，需要一个桥接机制。可以称为 Airalogy Source Bridge，也可以在实现层称为 Connector 或 Compatibility Layer。

Source Bridge 的职责不只是把外部系统中的对象暴露为 Airalogy 可理解的引用，还要为这些对象进入 Airalogy Record 体系提供标准化入口。它首先保留原始对象的来源、版本、权限和读取方式，然后让 Airalogy Protocol 在这些来源对象之上执行校验、抽取、确认、结构化和来源链记录，最终形成受 Protocol 约束的 Airalogy Record。

```json
{
  "airalogy_ref": "airalogy.ref.feishu.message.msg_xxx",
  "source": {
    "system": "feishu",
    "workspace_id": "tenant_xxx",
    "object_type": "message",
    "external_id": "msg_xxx"
  },
  "capabilities": ["read", "index", "resolve", "watch"],
  "content_hash": "sha256:...",
  "observed_at": "2026-05-30T10:00:00+08:00"
}
```

这个引用不要求 Airalogy 保存完整消息正文，但它必须说明数据是什么、在哪里、如何读取、是什么版本、是否可监听变化、是否可被 Protocol 作为输入，以及是否可以被转化为 Airalogy Record。

更准确地说，Source Bridge 解决的是“来源对象如何进入 Airalogy”的问题；Recordization Protocol 解决的是“来源对象如何在 Protocol 约束下成为 Airalogy Record”的问题。二者组合后，外部数据进入主脑的标准路径可以是：

```text
外部原始数据
  -> Source Bridge 暴露 SourceObjectRef 和可处理数据视图
  -> Recordization Protocol 执行 schema 校验、字段抽取、语义解释和人工确认
  -> 生成 Airalogy Record 或 Record 化视图
  -> 主脑以统一 Record 语义调用这些数据
```

这里的转化不意味着覆盖或丢弃原始数据。原始消息、邮件、文件、会议和仪器数据仍保留在原系统或 Raw Data Store 中；Airalogy Record 是在来源链、Protocol、版本、权限和确认机制约束下形成的标准化语义层。对于高价值、长期使用或需要审计的数据，可以物化为 Airalogy Record；对于临时查询或体量很大的数据，也可以先生成 Record 化视图，让主脑仍然以 Record 形式理解和使用。

与 Source Bridge 对应，Resource Bridge 负责相反方向：把 Airalogy 中经过 Protocol 约束的 Action Record 发送到外部资源，并把执行状态和结果带回 Airalogy。Source Bridge 让外部数据进入主脑可理解的 Record 语义；Resource Bridge 让主脑产生的 Record 化行动进入外部软件、设备、仪器、机器人或云服务。二者可以由同一个 connector 实现，也可以由不同 connector 实现，但都应当遵循 Protocol、权限、确认和来源链约束。

不同系统可以提供不同级别的兼容能力：

- Level 0: Export-only。只能通过导出文件接入，例如本地聊天记录导出、邮件归档包、仪器导出的 CSV。
- Level 1: Reference。Airalogy 只保存外部引用和基础元数据，外部系统仍是唯一事实来源。
- Level 2: Index。Airalogy 可以读取内容并建立全文索引、向量索引、关系索引和时间线。
- Level 3: Mirror。Airalogy 可以保存一份规范化副本，用于离线分析、版本冻结和跨系统查询，但原系统仍可保留事实来源地位。
- Level 4: Bidirectional。Airalogy 可以将任务、标注、摘要、状态或派生结果写回外部系统。

飞书、Slack、Gmail 等开放 API 较完整的系统可以逐步达到 Level 3 或 Level 4。微信个人聊天、封闭仪器软件或本地文件系统可能只能从 Level 0 或 Level 1 起步。这个分级让 Airalogy 可以兼容现实世界的不完整接口，而不是依赖所有系统一次性开放。

## Bridge 标准接口

每个 bridge connector 不需要实现全部能力，可以只支持 Source Bridge、只支持 Resource Bridge，或同时支持二者。但它们应尽量向同一组接口靠拢：

```text
list_objects(query)
resolve(ref)
get_metadata(ref)
get_content(ref)
prepare_protocol_input(ref, protocol_id)
recordize(ref, protocol_id)
watch_changes(cursor)
search(query)
write_back(ref, patch)
prepare_action(record, protocol_id)
execute_action(record)
get_execution_status(execution_ref)
collect_execution_result(execution_ref)
```

其中 `resolve(ref)` 是最基础的接口，`prepare_protocol_input(ref, protocol_id)` 和 `recordize(ref, protocol_id)` 把桥接能力推进到 Airalogy Record 层；`prepare_action(record, protocol_id)`、`execute_action(record)`、`get_execution_status(execution_ref)` 和 `collect_execution_result(execution_ref)` 则面向 Resource Bridge，把 Action Record 推送到外部资源并回收结果。Protocol、搜索系统、agent 和 UI 无需关心底层系统是飞书、微信、邮箱、文件系统、LIMS 还是实验设备。它们只需请求 Airalogy 解析某个 ref，或通过指定 Protocol 执行某个 Action Record。

## Protocol 的角色

在这个架构下，Protocol 不是所有数据的存储格式，而是语义合同、交互标准、原生采集入口和自动化运行时。主脑与外部软件、工具、设备、自动化系统和人的桥接，本质上可以抽象为一系列 Protocol 所定义的交互协议。Protocol 定义某类交互如何发生：输入是什么、输出是什么、如何调用、如何确认、如何校验、权限边界是什么、执行状态如何反馈、结果如何进入来源链。Airalogy Record 则是这些交互中的数据单元：它可以是输入、观察、行动方案、状态更新、执行结果、人工确认或派生结论，并带有明确的 `protocol_id`、`protocol_version`、schema、来源链、时间、执行者或确认者信息。

Protocol 的输出性质取决于它承担的角色。如果 Protocol 读取外部聊天、邮件、会议或文件并进行摘要、抽取、判断和整理，输出通常是派生对象，其中也可以包括基于来源证据生成的派生 Airalogy Record。如果 Protocol 的主要动作是直接记录实验步骤、观察结果、表单填写、设备读数、人工确认或组织决策，那么输出就是原生 Airalogy Record。也就是说，Protocol 不必承载所有数据，但 Airalogy Record 一定应当和 Protocol 绑定。

对于主脑，Protocol 是把智能意图落到可执行流程的关键媒介。主脑可以提出目标和计划，但真正进入系统执行时，需要 Protocol 明确输入、输出、约束、验证方式、可调用工具和人工确认点。Protocol 使主脑不只是“想”，而是能够以可追溯、可复现、可审计的方式“做”。

更进一步，Protocol 可以被理解为主脑的行动空间定义。每一种 Action Protocol 都代表主脑可以采取的一类行动，例如发送消息、创建任务、安排会议、调用软件流程、启动设备、执行实验步骤、触发机器人动作或向外部系统写入状态。主脑基于目标、上下文和约束选择某个 Protocol，并在该 Protocol 下生成一个具体的 Action Record。这个 Record 记录行动的目标、参数、对象、资源、权限、确认状态、执行计划和预期结果，是一次具体行动的结构化方案。

当 Action Record 需要作用于外部世界时，它可以通过对应的 Resource Bridge 或 connector 发送到外部资源，例如飞书、邮件系统、LIMS、自动化设备、实验仪器、业务软件、机器人或云服务。外部资源执行后，执行状态、日志、结果、异常和人工确认再回流为新的 Airalogy Record 或 AssistantExecution。这样，主脑的能力不是不受约束地控制外部世界，而是在 Protocol、权限、确认、来源链和执行记录约束下，调度和控制任何已经接入且被授权的资源。

从主脑的视角看，Source Bridge 和 Protocol 的组合让各种异构数据都可以被“Record 化”。主脑不需要直接理解飞书消息模型、邮件 MIME 结构、会议转写格式或仪器导出格式，而是调用经过 Protocol 约束后的 Airalogy Record 或 Record 化视图。这样，所有数据在智能运行时中都可以拥有统一的来源链、schema、权限、版本和可验证语义。

例如一个聊天行动项抽取 Protocol 可以声明：

```text
Protocol: chat_action_item_extraction

inputs:
- chat_thread: ChatThreadRef
- time_range: TimeRange
- participant_filter: list[PersonRef]

outputs:
- summary: AiralogyMarkdown
- decisions: list[Decision]
- action_items: list[Task]
- evidence_messages: list[MessageRef]
```

执行流程是：

```text
外部聊天系统
  -> Source Bridge 暴露 ChatThreadRef 和 MessageRef
  -> Protocol Engine 读取指定消息范围
  -> Protocol 执行摘要、抽取、校验和人工确认
  -> 生成 Airalogy Record、Task、Decision、Annotation 或 Document
  -> 每个输出对象记录 derived_from、protocol_id、protocol_version 和 source_hash
```

这样，聊天本体仍然保持消息系统的专门结构，Airalogy 则拥有可审计的派生结果和跨系统关系。

## 统一对象信封

为了让不同来源的数据进入同一语义网络，可以为 Airalogy 管理的对象定义统一信封：

```json
{
  "airalogy_object_id": "airalogy.id.message.xxxx",
  "object_type": "message",
  "schema_version": "1.0.0",
  "source_ref": {
    "system": "feishu",
    "external_id": "msg_xxx"
  },
  "created_at": "2026-05-30T09:30:00+08:00",
  "observed_at": "2026-05-30T10:00:00+08:00",
  "actor_ids": ["airalogy.id.person.alice"],
  "raw_refs": ["airalogy.id.file.raw_msg_xxx.json"],
  "links": [
    {
      "rel": "belongs_to",
      "target": "airalogy.id.thread.xxxx"
    }
  ],
  "payload": {
    "kind": "text",
    "text": "今天的方案需要调整。"
  }
}
```

统一信封解决的是跨对象治理问题：ID、类型、版本、来源、原始引用、关系和审计字段保持一致。`payload` 则允许不同对象保留自己的专门结构。

## 来源链、Airalogy Record 与派生对象

AI-ready 数据和智能中枢必须把 Airalogy Record、派生结果和原始证据连接起来。任何 AI 摘要、任务、决策、实验判断或结构化抽取结果都应当记录来源链；任何原生采集形成的 Airalogy Record 也应当记录对应的 Protocol、执行者、确认者和采集时间：

```json
{
  "airalogy_object_id": "airalogy.id.record.xxxx.v.1",
  "object_type": "record",
  "derived_from": [
    "airalogy.ref.feishu.message.msg_001",
    "airalogy.ref.feishu.message.msg_002",
    "airalogy.id.file.xxxx.pdf"
  ],
  "protocol_id": "chat_action_item_extraction",
  "protocol_version": "1.0.0",
  "source_hash": "sha256:..."
}
```

这使 AI 输出不只是“看起来合理”，而是能回答：结论来自哪里、依据是什么、使用了哪个 Protocol、是否可复现、是否可人工复核。

## Airalogy 的独特优势

Airalogy 的优势不在于比专业系统更懂聊天、邮件、会议或仪器数据，而在于它可以提供一个跨场景的语义控制面：

- 它以 Airalogy Record 承载基于 Protocol 的原生数据记录，这是 Airalogy 区别于普通数据桥接平台的核心特色。
- 它用 Protocol 显式描述数据生成、采集、校验、转换、派生、交互和行动规则。
- 它用 AIMD 连接人类可读文档和机器可读字段。
- 它用 Airalogy Record、File、ID、hash、版本和 metadata 承载可审计的结构化结果。
- 它可以通过 Source Bridge 和 Resource Bridge 协同外部专业系统，而不是替代它们。
- 它可以把原始数据变成任务、决策、摘要、实验记录、证据链和后续工作流。
- 它可以为主脑提供长期记忆、可调用上下文、工具边界、执行历史和证据追踪。

因此，Airalogy 更像是 universal data and intelligence orchestration layer，而不是 universal database。它让不同系统的数据从“被保存”提升为“可被 AI 正确使用”，并进一步成为主脑式智能可以持续推理、调度和行动的基础。

## 核心平台概念

Airalogy 的通用框架需要一组稳定的平台概念作为共同语言：

- `AiralogyObjectId`: 所有 Airalogy 管理对象的统一 ID。
- `AiralogyRecord`: 基于 Protocol 记录、校验、生成或交换的数据对象，是主脑与数据、工具、设备、人员和外部系统交互时传递和确认的结构化数据单元；可以是原生采集形成的 source record，也可以是基于来源证据形成的 derived record。
- `SourceObjectRef`: 外部系统对象的统一引用。
- `ConnectorCapability`: 外部系统兼容能力分级。
- `AiralogyObjectEnvelope`: 统一对象信封。
- `Protocol`: 主脑与人、数据、软件、设备和自动化资源交互的标准协议，定义输入、输出、约束、权限、确认、执行方式和反馈机制。
- `ProtocolInput` / `ProtocolOutput`: Protocol 与外部对象协同的输入输出合同。
- `RecordizationProtocol`: 将外部来源对象或内部原始对象转化为 Airalogy Record 或 Record 化视图的 Protocol。
- `RecordView`: 面向主脑和 agent 的 Record 化视图，在不必物化完整 Record 时提供统一 Record 语义。
- `ActionProtocol`: 定义主脑可以采取的一类行动，包括输入约束、目标资源、权限边界、确认点和执行方式。
- `ActionRecord`: 在某个 Action Protocol 下生成的一次具体行动方案，记录目标、参数、对象、资源、确认状态、执行计划和预期结果。
- `ResourceBridge`: 将 Action Record 发送到外部软件、设备、仪器、机器人或云服务，并回收执行状态和结果的桥接能力。
- `DerivedObject`: 由 Protocol、AI 或人工流程产生的派生对象。
- `Lineage`: 原始来源、Protocol、版本、hash 和人工确认记录。
- `Relation`: 任意对象之间的 typed link。
- `PermissionAwareContext`: 面向 AI agent 的权限感知上下文装配。
- `MasterbrainRuntime`: 面向长期目标、上下文装配、工具调度、执行反馈和复盘的主脑运行时。
- `MasterbrainIntent`: 记录人、组织或 AI 设定的高层目标、策略、约束和价值边界。
- `AssistantExecution`: 记录人、软件工具、外部系统、自动化设备或 AI agent 执行具体任务的过程和结果。

这些概念构成 Airalogy 作为通用框架需要长期保持稳定的语义骨架。只要这套骨架稳定，实验、邮件、聊天、会议、文件、任务、网页、代码和业务系统数据都可以以不同方式接入，并在主脑中以统一 Record 语义被使用。

## 平台能力体系

围绕这些核心概念，Airalogy 将呈现为一组相互连接的平台能力：

- Protocol-backed Record 能力：通过 Airalogy Protocol 原生记录、校验或生成 `AiralogyRecord`，让 Airalogy 本身成为重要数据的出生地和事实来源之一。
- Protocol 交互能力：以 Protocol 定义主脑与人、数据、软件、设备和自动化工具的交互标准，以 Record 承载每次交互的输入、输出、状态、结果和确认。
- 统一对象与引用能力：通过 `AiralogyObjectEnvelope`、`SourceObjectRef`、`File`、`Message`、`Thread`、`Document`、`Task`、`Annotation` 等对象，让内部数据和外部系统对象进入同一个语义网络。
- Source / Resource Bridge 能力：通过 connector 规范和 capability 分级接入文件系统、导出包、邮件、飞书、Slack、LIMS、业务系统、设备、仪器和其他专业软件；同时支持外部数据进入 Airalogy，以及 Airalogy 行动进入外部资源。
- Record 化能力：通过 Recordization Protocol 将来源对象转化为 Airalogy Record 或 Record 化视图，让主脑能够以统一 Record 语义使用异构数据。
- 行动执行能力：通过 Action Protocol 和 Action Record 把主脑意图转化为可审计的行动方案，并通过 Resource Bridge 发送到外部软件、设备、仪器、机器人或云服务。
- 智能上下文能力：通过关系、时间线、全文索引、向量索引和权限感知上下文装配，让 agent 基于 Airalogy 的对象网络工作。
- 跨系统协同能力：让 AI 或 Protocol 生成的任务、标注、摘要和状态按权限写回外部系统，形成跨系统工作流闭环。
- 主脑运行时能力：让人、组织或 AI 设定目标，Airalogy 主脑基于统一数据网络进行规划、推理、调度和复盘，多个 Assistant 在其调度下执行任务并持续反馈真实世界结果。

这些能力共同构成 Airalogy 作为数据和智能中枢的能力体系。它们支撑 Airalogy 从数据记录工具、跨系统桥接层，进一步演化为主脑可以运行其上的通用智能基础设施。

## 结论

如果未来存在一个让 AI 随时 ready 的数据基础设施，它不会只是一个更大的网盘、数据库、聊天软件或向量库。它需要保存或引用原始数据，也需要理解数据的来源、结构、关系、权限、版本和处理规则。

Airalogy 将建设这层语义和智能基础设施：以 Airalogy Record 承载基于 Protocol 的原生数据记录和交互数据，同时让专业系统继续负责各自最擅长的存储和交互。Airalogy 通过 Source Bridge、Resource Bridge、统一对象、Protocol、来源链和派生对象，把分散数据组织成 AI 可以直接使用、可以追溯、可以验证、可以执行的数据和智能中枢。

更长远地说，Airalogy 应当成为主脑的平台形态。目标和使命可以来自人、组织、AI 或它们的混合结构；Airalogy 则组织数据、理解世界、形成计划、调度助手、追踪证据并持续复盘。这样，Airalogy 不只是让 AI “有数据可读”，而是让 AI 有条件成为真正能够协同生产、生活、科学活动甚至组织经营的主脑系统。
